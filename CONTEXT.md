# Mathilde — Adaptive Math Learning

An adaptive math learning app for a child with ADHD attending Austrian Gymnasium. Two personas interact with the system: a Curator who controls curriculum and reviews content, and a Learner who works through sessions on an iPad.

## Language

### Personas

**Curator**:
The parent who controls curriculum, reviews sessions, and monitors progress via CLI.
_Avoid_: Admin, teacher, parent (in code context)

**Learner**:
The child who interacts with the app during practice sessions.
_Avoid_: Student, user, kid

### Learner State

**Profile**:
The Curator-managed configuration for a Learner — mission and preferences. Backend-owned, stored at `users/{uid}/profile/main`.
_Avoid_: Settings, config, user record

**Progress**:
The Learner's earned XP and level. Shell-owned, stored at `users/{uid}/progress/main`. Can never decrease.
_Avoid_: Stats, score, profile (which refers to the Curator-managed configuration)

### Pedagogy

**Mission**:
The reason the Learner is studying a subject — a concrete real-world goal that grounds all teaching decisions.
_Avoid_: Goal, objective, purpose

**Zone of Proximal Development (ZPD)**:
The sweet spot between what the Learner already knows and what's just beyond reach — where learning happens.
_Avoid_: Difficulty level, skill level

**Storage Strength**:
Long-term retention of knowledge, the real goal of learning. Contrasted with fluency (in-the-moment retrieval that fades).
_Avoid_: Memory, retention

**Hint Chain**:
A sequence of 2-3 progressive hints per exercise that decrease in difficulty. The Learner never fails — she arrives at understanding with more or less scaffolding.
_Avoid_: Help, clue, tip

**Learning Record**:
A record of what the Learner knows, doesn't know, or has misconceptions about. Drives ZPD calculation and session planning. Contains both structured fields (status, interval) and narrative (Claude's analysis).
_Avoid_: Progress record, skill record, log

### Curriculum

**Topic**:
A curriculum unit the Learner encounters in school (e.g., Bruchrechnung, Gleichungen). Added by the Curator when the school starts covering it.
_Avoid_: Subject, module, unit

**Concept**:
The atomic learning unit within a Topic. One Concept maps to one Learn session. Has prerequisites linking to other Concepts.
_Avoid_: Lesson, skill, objective

### Sessions

**Session**:
A planned learning experience generated by the Planner and approved by the Curator. Contains Claude-generated HTML with exercises, explanations, hints, and feedback. Completable in 10-15 minutes.
_Avoid_: Lesson (ambiguous with teach skill), exercise set, assignment

**Session Type**:
The kind of session the Planner generates: `learn` (new concept), `drill` (practice known material), `quiz` (assessment), `review` (re-teach struggled concept), `exam-prep` (Schularbeit simulation).
_Avoid_: Mode, category

**Exercise**:
A single question or task within a Session. Either pre-evaluated (answer known at generation time) or Claude-evaluated (handwriting, free-form).
_Avoid_: Question, problem, task (too generic)

**Attempt**:
The Learner's answer to an Exercise — a selection, typed number, or handwriting image.
_Avoid_: Answer, response, submission

**Evaluation**:
The assessment of an Attempt — either instant (pre-evaluated) or async via Claude Vision (handwriting).
_Avoid_: Grading, scoring, checking

### System

**Planner**:
The backend component that reads Learning Records and session results, calculates ZPD, calls Claude to generate Sessions, and writes them to Firestore for Curator review.
_Avoid_: Scheduler, generator, engine

**Reactor**:
The backend component that watches Firestore for events (handwriting evaluations, session completions) and processes them.
_Avoid_: Listener, watcher, handler

**Bridge**:
The `window.mathilde.*` JavaScript API that the SPA shell exposes for Claude-generated HTML to communicate session events back to the app.
_Avoid_: API, interface, callback

**Shell**:
The static SPA that handles auth, navigation, progress display, and renders Session HTML. Does not contain exercise logic — that lives in the generated HTML.
_Avoid_: App, frontend, client

### Data Principles

**Document Ownership**:
Every Firestore document has exactly one owning component that performs full-replace writes. Other components may read but never write. This eliminates merge conflicts and partial updates. Session documents are the exception — status transitions cross component boundaries and use conditional writes (UpdateTime precondition) for safety.

**Document Identity**:
Documents with natural keys use deterministic IDs: Topics use `slugify(name)`, Learning Records use `conceptId`, Profile and Progress use `main` (singletons). Documents without natural keys (Sessions, Evaluations) use auto-generated IDs.

## Relationships

- A **Curator** adds **Topics** and approves **Sessions**
- A **Topic** contains one or more **Concepts** with prerequisite links between them
- The **Planner** generates **Sessions** based on **Learning Records** and the **Mission**
- A **Session** contains multiple **Exercises**, each with a **Hint Chain**
- The **Learner** produces **Attempts** for each **Exercise**
- An **Attempt** receives an **Evaluation** (instant or via **Reactor**)
- **Evaluations** feed back into **Learning Records**, which drive the next **Planner** cycle
- The **Shell** renders **Session** HTML and exposes the **Bridge** for communication

## Example dialogue

> **Dev:** "When the **Learner** completes a **Session**, does the **Planner** generate the next one immediately?"
> **Domain expert:** "No — the **Reactor** first sends the results to Claude for analysis, which updates the **Learning Records**. The **Planner** then uses the updated records to generate the next **Session**. The **Curator** still has to approve it before the **Learner** sees it."

> **Dev:** "If the **Learner** uses all hints in a **Hint Chain**, is that a failed **Exercise**?"
> **Domain expert:** "There is no failure. She always earns XP. But the **Learning Record** for that **Concept** gets a narrative noting she needed full scaffolding — the **Planner** will schedule a **Review** session."

## Flagged ambiguities

- "lesson" — the teach skill calls its HTML output a "lesson." We use **Session** to avoid confusion with school lessons (Unterrichtsstunde). The teach skill remains a pedagogical reference, not a runtime component.
- "static" vs "dynamic" — Session content is generated by Claude considering the Learner's state (adaptive), but the output is a fixed HTML artifact (static). "Static" refers to the artifact, not the intelligence behind it.
- "backend" — refers to the Go process running on the home machine that contains both the **Planner** and the **Reactor**. Not a web server — the Shell never talks to it directly. All communication goes through Firestore.
